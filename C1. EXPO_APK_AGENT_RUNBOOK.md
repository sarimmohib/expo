# AI Agent Runbook: Wrapping a Next.js PWA into an Installable Android APK with Expo

> **Audience:** AI coding agents (Claude, GPT, Gemini, etc.) that have just been handed a Next.js PWA source tree and asked to produce an installable `.apk`.
>
> **What this is:** A field-tested, opinionated playbook derived from shipping one such app (CLIL Engine — Next.js 16 + Turbopack, Expo SDK 53, React Native 0.79, react-native-webview 13). Every error, fix, and gotcha documented here was hit for real. Follow it linearly the first time; treat it as a reference manual afterward.
>
> **What this is NOT:** A general Expo tutorial. It assumes you already know what Expo / React Native / Gradle / Android SDK are, and focuses on the *non-obvious* stuff that bites you when you wrap a server-rendered PWA inside a native WebView shell.

---

## 0. TL;DR — the 30-second version

1. Convert the Next.js app to `output: "export"` static HTML.
2. Delete any `/api/*` routes — they can't run in a static export. Either delete the feature or reimplement it client-side (most third-party APIs send permissive CORS headers, so this is usually possible).
3. Bundle the entire `out/` directory into a single TypeScript data module (`web-bundle.ts`) — text files as UTF-8, binary as base64 — and rewrite `/_next/...` → `./_next/...` inside `.html` files.
4. Create a minimal Expo app with a single `<WebView>` that unpacks the bundle to `FileSystem.documentDirectory` on first launch and loads `index.html` with `allowFileAccess` + `allowUniversalAccessFromFileURLs`.
5. Install Android SDK + NDK 27 + CMake 3.22.1, run `expo prebuild`, run `./gradlew assembleRelease`.
6. Patch the Turbopack `getAssetPrefix` check or React will never hydrate (see §8.2).
7. Ship the APK.

## 2. Toolchain prerequisites

Check these before you start. Missing any of them costs hours.

| Tool | Required version | How to verify |
|---|---|---|
| Node.js | ≥ 20 (24 is fine) | `node --version` |
| Java JDK | **21** (NOT 17) | `java -version` — needs to say `21.x.x` |
| Android SDK cmdline-tools | latest | `sdkmanager --version` |
| Android platform-tools | latest | `adb --version` |
| Android platforms | `android-35` | `ls $ANDROID_HOME/platforms` |
| Android build-tools | `35.0.0` | `ls $ANDROID_HOME/build-tools` |
| Android NDK | **`27.1.12297006`** (exact version!) | `ls $ANDROID_HOME/ndk` |
| CMake | `3.22.1` (exact version!) | `ls $ANDROID_HOME/cmake` |
| Python | ≥ 3.10 | `python3 --version` |

**Do NOT skip the NDK/CMake exact-version requirement.** Expo SDK 53 / RN 0.79 hardcodes `27.1.12297006` and `3.22.1` in their gradle config. Other versions will cause CMake to fail with cryptic errors like `Cannot specify link libraries for target "react_codegen_RNCWebViewSpec"`.

### 2.2 Java home gotcha

`readlink -f $(which java)` returns `/usr/lib/jvm/java-21-openjdk-amd64/bin/java`. Gradle wants the **parent of `bin/`**, i.e. `/usr/lib/jvm/java-21-openjdk-amd64`. If you pass the wrong path, gradle fails with `JAVA_HOME is set to an invalid directory`.

## 3. Intake: inspecting the Next.js source

Before changing anything, audit the source for things that **cannot survive a static export**.

```bash
cd /home/z/my-project/work
# Find every API route — these will break the static export
rg --files src/app/api 2>/dev/null
# Find every server-side fetch — these need to move client-side
rg "export const runtime|export const dynamic|fetch\(" src/app/api 2>/dev/null
# Find usage of those routes from the client
rg "fetch\(['\"]/api/" src/ --type ts --type tsx
```

**Decision tree for each API route:**
- Does it just proxy a third-party API (GitHub, Stripe, etc.)? **→** Delete it; the client can call the third party directly (most send permissive CORS headers).
- Does it read a database or filesystem? **→** You have a real architectural problem. Either ship a backend separately, or store the data in `localStorage` / `IndexedDB` / `AsyncStorage` instead.
- Does it do CPU-heavy work that can't run client-side? **→** Same problem; ship a backend.
- Does it just expose environment variables to the client? **→** Use `NEXT_PUBLIC_*` env vars instead, baked into the static bundle.

For the CLIL Engine project, the `/api/github` route was a CORS-proxy for GitHub. Both `api.github.com` and `raw.githubusercontent.com` send `Access-Control-Allow-Origin: *`, so the proxy was unnecessary — I deleted the route and reimplemented the fetch client-side.

## 4. Step 1 — Convert Next.js to static export

Edit `next.config.ts`:

```ts
import type { NextConfig } from "next";

const nextConfig: NextConfig = {
  output: "export",          // <- the magic line
  images: { unoptimized: true },  // required: no image optimizer in static export
  trailingSlash: true,       // optional but helps with file:// origins
  typescript: { ignoreBuildErrors: true },   // be lazy — your job is to ship, not refactor
  eslint: { ignoreDuringBuilds: true },
  reactStrictMode: false,
};

export default nextConfig;
```

**Then delete the API routes:**
```bash
rm -rf src/app/api
```

**Build:**
```bash
bun install  # or npm install
bun run build
# Output lands in ./out/
```

**Verify success:**
- `out/index.html` exists
- `out/_next/static/chunks/*.js` exists (lots of them)
- `out/_next/static/css/*.css` exists
- `out/manifest.json` exists (if the PWA had one)

If the build fails, the two most common reasons are:
1. Some page uses `export const dynamic = 'force-dynamic'` — remove it.
2. Some page uses `fetch()` at module scope during SSR — wrap it in `useEffect` or `if (typeof window !== 'undefined')`.

## 5. Step 2 — Bundle the static export as a TypeScript module

### 5.1 Why a data module (not Metro assets)

You might be tempted to drop `out/` into `expo-app/assets/web/` and use `expo-asset`. **Don't.** Metro (the React Native bundler) treats `.js` files as JS modules — it'll try to parse your `_next/static/chunks/*.js` as source code, which will fail catastrophically.

The robust approach: serialize the entire export as a TS object literal. Text files as UTF-8 strings, binary files (PNG, woff2) as base64 strings. Metro bundles TS literals without complaining.

### 5.2 The build-web-bundle.py script

Save this as `/home/z/my-project/scripts/build-web-bundle.py`. It reads `work/out/` and writes `expo-app/src/web-bundle.ts`.

```python
#!/usr/bin/env python3
"""
Build a TypeScript data module from a Next.js static export.

Reads:  /home/z/my-project/work/out/
Writes: /home/z/my-project/expo-app/src/web-bundle.ts

Why a data module instead of bundling assets/web/ via expo-asset:
  Metro parses .js files as JS modules. Renaming them to .txt and rewriting
  references is fragile. A single self-contained data module is simple to
  bundle, simple to inspect, and works in Expo Go without native modules.

Also rewrites absolute URL references (/_next/..., /index.html) to relative
(./_next/..., ./index.html) so the PWA loads correctly from a file:// origin.
"""

import base64
import json
import re
from pathlib import Path

SOURCE_DIR = Path("/home/z/my-project/work/out")
OUTPUT_FILE = Path("/home/z/my-project/expo-app/src/web-bundle.ts")

# Skip these Next.js build artefacts — they waste space and aren't needed at runtime.
SKIP_PATTERNS = [
    re.compile(r"^__next\."),         # build-manifest.txt etc.
    re.compile(r"\.txt$"),            # pre-rendered text exports
    re.compile(r"^404($|/)"),         # 404 page
    re.compile(r"^_not-found($|/)"),
    re.compile(r"^robots\.txt$"),
]


def should_skip(rel_path: str) -> bool:
    return any(p.search(rel_path) for p in SKIP_PATTERNS)


def rewrite_paths(content: str) -> str:
    """Rewrite absolute URL references in HTML so the file loads from file://."""
    content = re.sub(r'((?:src|href)=["\'])/_next/', r"\1./_next/", content)
    content = re.sub(r'((?:src|href)=["\'])/(?!/)', r"\1./", content)
    content = re.sub(r'url\(["\']?/_next/', "url(./_next/", content)
    content = re.sub(r'"/_next/', '"./_next/', content)
    return content


def main() -> None:
    if not SOURCE_DIR.exists():
        raise SystemExit(f"Source directory not found: {SOURCE_DIR}")

    files = {}
    total_bytes = 0
    skipped = 0

    for root, dirs, filenames in SOURCE_DIR.walk():  # type: ignore[attr-defined]
        dirs[:] = [d for d in dirs if not d.startswith(".") and not d.startswith("__")]
        for filename in filenames:
            full_path = root / filename
            rel_path = str(full_path.relative_to(SOURCE_DIR)).replace("\\", "/")
            if should_skip(rel_path):
                skipped += 1
                continue

            data = full_path.read_bytes()
            total_bytes += len(data)

            try:
                text = data.decode("utf-8")
                if rel_path.endswith(".html") or rel_path.endswith(".htm"):
                    text = rewrite_paths(text)
                files[rel_path] = {"type": "text", "content": text}
            except UnicodeDecodeError:
                files[rel_path] = {
                    "type": "base64",
                    "content": base64.b64encode(data).decode("ascii"),
                }

    OUTPUT_FILE.parent.mkdir(parents=True, exist_ok=True)
    with OUTPUT_FILE.open("w", encoding="utf-8") as f:
        f.write("// AUTO-GENERATED — DO NOT EDIT.\n")
        f.write("// Produced by scripts/build-web-bundle.py from the Next.js static export.\n\n")
        f.write("export interface WebBundleFile {\n  type: 'text' | 'base64';\n  content: string;\n}\n\n")
        f.write("export const WEB_BUNDLE: Record<string, WebBundleFile> = ")
        f.write(json.dumps(files, ensure_ascii=False, separators=(",", ":")))
        f.write(";\n")

    print(f"Wrote {OUTPUT_FILE}")
    print(f"  Files: {len(files)} (skipped {skipped})")
    print(f"  Source bytes: {total_bytes:,}")
    print(f"  Output size: {OUTPUT_FILE.stat().st_size:,} bytes")


if __name__ == "__main__":
    main()
```

Run it:
```bash
python3 /home/z/my-project/scripts/build-web-bundle.py
```

Expected output: ~3.5 MB source tree → ~3.7 MB `web-bundle.ts` (base64 inflation).

### 6.4 App.tsx — the WebView shell

This is the heart of the wrapper. It does four things:
1. Unpacks `WEB_BUNDLE` to `FileSystem.documentDirectory/web/` on first launch (or when `BUNDLE_VERSION` changes).
2. Loads `index.html` in a `<WebView>` with file:// access enabled.
3. Captures JS errors via an injected bootstrap script and surfaces them in a tappable on-device console (because users can't use `chrome://inspect`).
4. Wires the hardware back button to WebView history.

The full file is ~400 lines. Read it in `/home/z/my-project/expo-app/App.tsx`. Key snippets:

**Bundle unpacking:**
```ts
async function ensureWebBundle(onProgress: (p: number) => void) {
  // Skip if version marker matches
  const marker = await FileSystem.getInfoAsync(VERSION_MARKER);
  const index = await FileSystem.getInfoAsync(INDEX_FILE);
  if (marker.exists && index.exists) {
    const v = await FileSystem.readAsStringAsync(VERSION_MARKER);
    if (v === BUNDLE_VERSION) return;
  }

  // Wipe + recreate the directory
  await FileSystem.deleteAsync(WEB_DIR, { idempotent: true });
  await FileSystem.makeDirectoryAsync(WEB_DIR, { intermediates: true });

  // Write every file
  for (const [relPath, file] of Object.entries(WEB_BUNDLE)) {
    const dest = `${WEB_DIR}${relPath}`;
    // Ensure parent dir exists
    const lastSlash = dest.lastIndexOf("/");
    if (lastSlash > WEB_DIR.length - 1) {
      await FileSystem.makeDirectoryAsync(dest.substring(0, lastSlash), { intermediates: true });
    }
    await FileSystem.writeAsStringAsync(
      dest,
      file.content,
      { encoding: file.type === "text" ? FileSystem.EncodingType.UTF8 : FileSystem.EncodingType.Base64 }
    );
  }

  // Stamp the version
  await FileSystem.writeAsStringAsync(VERSION_MARKER, BUNDLE_VERSION, {
    encoding: FileSystem.EncodingType.UTF8,
  });
}
```

**The injected bootstrap (for on-device error capture):**
```js
const JS_BRIDGE_BOOTSTRAP = `
(function(){
  function send(kind, text){
    try {
      var t = String(text);
      if (t.length > 4000) t = t.slice(0, 4000) + '…[truncated]';
      window.ReactNativeWebView && window.ReactNativeWebView.postMessage(
        JSON.stringify({kind: kind, text: t})
      );
    } catch(e){}
  }
  window.addEventListener('error', function(e){
    var msg = e.message || 'Error';
    if (e.filename) msg += ' @ ' + e.filename.split('/').pop() + ':' + e.lineno;
    if (e.error && e.error.stack) msg += '\\n' + e.error.stack;
    send('clil-error', msg);
  });
  window.addEventListener('unhandledrejection', function(e){
    var r = e.reason;
    var msg = 'Promise rejection: ' + (r && r.message ? r.message : String(r));
    if (r && r.stack) msg += '\\n' + r.stack;
    send('clil-error', msg);
  });
  var origErr = console.error;
  console.error = function(){
    try {
      var parts = [];
      for (var i = 0; i < arguments.length; i++) {
        var a = arguments[i];
        if (a instanceof Error) parts.push(a.stack || a.message);
        else if (typeof a === 'object') {
          try { parts.push(JSON.stringify(a)); } catch(_) { parts.push(String(a)); }
        } else parts.push(String(a));
      }
      send('clil-console-error', parts.join(' '));
    } catch(_){}
    return origErr.apply(this, arguments);
  };
  // Defensive: rewrite stray /_next/ fetches
  var origFetch = window.fetch;
  if (origFetch) {
    window.fetch = function(input, init){
      if (typeof input === 'string' && input.indexOf('/_next/') === 0
          && input.indexOf('./_next/') !== 0) {
        input = '.' + input;
      }
      return origFetch.call(this, input, init);
    };
  }
  var origOpen = XMLHttpRequest.prototype.open;
  XMLHttpRequest.prototype.open = function(method, url){
    if (typeof url === 'string' && url.indexOf('/_next/') === 0
        && url.indexOf('./_next/') !== 0) {
      url = '.' + url;
    }
    return origOpen.apply(this, arguments);
  };
})();
true;
`;
```

**On-device error console modal:**

Build a `<Modal>` that shows ALL captured errors in a scrollable, monospace view with three buttons:
- **Copy all** — uses `expo-clipboard`'s `Clipboard.setStringAsync()` to copy the entire log to the system clipboard so the user can paste it into a chat with you.
- **Clear** — wipes the captured errors.
- **Close** — dismisses the modal.

This is essential because users on Android phones have no way to access `chrome://inspect`. The red banner at the bottom of the screen shows the latest error; tapping it opens the modal.

### 7.2 Gradle build

```bash
cd /home/z/my-project/expo-app/android
chmod +x gradlew
./gradlew :app:assembleRelease \
  -x lintVitalReportRelease \
  -x lintVitalAnalyzeRelease \
  -x lintVitalRelease \
  --no-daemon \
  --console=plain \
  --max-workers=2
```

**Flags explained:**
- `-x lintVital*` — skip lint. Lint is slow, eats disk, and fails on minor style issues that don't affect the APK.
- `--no-daemon` — don't leave a gradle daemon running (saves memory in sandboxed envs).
- `--max-workers=2` — limit parallelism. Default is `nproc`, which on a beefy machine can OOM.

**Expected duration:** 3–7 minutes for a cold build. Subsequent builds are faster (~1 min) if you don't `rm -rf app/build` and don't change the JS bundle.

**Expected output:**
```
app/build/outputs/apk/release/app-release.apk   (~60-70 MB)
```

## 8. The bugs you WILL hit

These are not theoretical. Every one of these was a real failure that cost me 10+ minutes. They are listed in the order you'll hit them.

### 8.1 `cp: cannot create directory '.next/standalone/.next/'`

**Cause:** The `build` script in `package.json` has post-build copy steps that assume `output: "standalone"`. After you switch to `output: "export"`, those steps fail.

**Fix:** Edit `package.json`:
```json
"build": "next build"
```
Remove the `&& cp -r ...` parts.

### 8.2 ⚠️ THE BIG ONE: Buttons don't work, only scrolling works

**Symptom:** APK installs and launches. The page renders. You can scroll it. But NO buttons, tabs, or interactive elements respond. Taps do nothing.

**Cause:** React never hydrated. The Turbopack runtime (`_next/static/chunks/turbopack-*.js`) hardcodes `let t="./_next/"` (after your path-rewriting) as its publicPath. Then `getAssetPrefix()` in `_next/static/chunks/771dedee3f5e1621.js` does:

```js
let { pathname } = new URL(document.currentScript.src);
if (pathname.indexOf("./_next/") === -1)
  throw new InvariantError(`Expected document.currentScript src to contain './_next/'`);
```

Under `file://` origin, `new URL('file:///data/.../web/_next/static/chunks/771.js').pathname` is `/data/.../web/_next/static/chunks/771.js` — which contains `_next/` but **never** `./_next/`. So the very first dynamic chunk load throws, and React hydration aborts.

**Fix:** Patch the indexOf check. Save as `/home/z/my-project/scripts/fix-asset-prefix-check.py`:

```python
#!/usr/bin/env python3
"""Patch the getAssetPrefix InvariantError in 771dedee3f5e1621.js."""
import json, re
from pathlib import Path

BUNDLE = Path("/home/z/my-project/expo-app/src/web-bundle.ts")
text = BUNDLE.read_text()

# Find the WEB_BUNDLE object literal (skipping the interface declaration)
m = re.search(r"WEB_BUNDLE[^=]*=\s*", text)
start = m.end()
i, depth, in_str, str_ch, escape = start, 0, False, None, False
while i < len(text):
    ch = text[i]
    if in_str:
        if escape: escape = False
        elif ch == "\\": escape = True
        elif ch == str_ch: in_str = False
    else:
        if ch in ('"', "'", "`"): in_str, str_ch = True, ch
        elif ch == "{": depth += 1
        elif ch == "}":
            depth -= 1
            if depth == 0: break
    i += 1

data = json.loads(text[start:i+1])
target = "_next/static/chunks/771dedee3f5e1621.js"  # hash may differ — search by content instead
# Robust: find the chunk that contains the invariant
for k, v in data.items():
    if k.endswith(".js") and "Expected document.currentScript src to contain" in v.get("content", ""):
        v["content"] = v["content"].replace('t.indexOf("./_next/")', 't.indexOf("_next/")')
        print(f"Patched: {k}")
        break

new_bundle_str = json.dumps(data, ensure_ascii=False, separators=(",", ":"))
# Re-find the object bounds after possible re-parsing
# (the index variables above are still valid since we didn't change text length much)
new_text = text[:start] + new_bundle_str + text[i+1:]
BUNDLE.write_text(new_text)
print(f"Wrote: {BUNDLE}")
```

Run it AFTER `build-web-bundle.py`:
```bash
python3 /home/z/my-project/scripts/build-web-bundle.py
python3 /home/z/my-project/scripts/fix-asset-prefix-check.py
```

**Bump `BUNDLE_VERSION` in App.tsx** so existing installs re-unpack the patched bundle.

### 8.9 Gradle builds "succeed" but no APK appears

**Cause:** The gradle daemon exited (due to OOM, signal, or `--no-daemon` shutdown) before finishing the assemble task, but the build log was swallowed by backgrounding.

**Fix:** Always check BOTH:
1. `find /home/z/my-project/expo-app/android/app/build/outputs -name "*.apk"`
2. The latest daemon log: `LATEST_LOG=$(ls -t /home/z/.gradle/daemon/8.13/*.log | head -1); grep -E "BUILD SUCCESSFUL|BUILD FAILED|FAILURE:" "$LATEST_LOG" | tail -10`

### 8.10 Background gradle processes die silently

**Cause:** Using `nohup ... &` in a non-interactive shell. The shell exits, killing the child.

**Fix:** Use `setsid` to fully detach:

Or run in foreground with a long tool timeout:
```bash
./gradlew :app:assembleRelease ... 2>&1 | tail -30
```
with `timeout: 600000` (10 min) on the Bash tool call.

### 8.11 The `expo start --tunnel` URL is unreachable from the phone

**Reality check:** In a sandboxed cloud environment, ngrok tunnels connect but the Expo dev server itself terminates within 30–60 seconds (memory limits, idle timeouts, container recycling). The tunnel URL dies with the process.

**Recommendation:** Don't promise the user an Expo Go URL. Build the APK instead. If they really want a dev URL, they should run `npx expo start --tunnel` from their own machine.

## 10. On-device debugging (because users can't use chrome://inspect)

This is the most important UX lesson: **your user is on a phone. They cannot open chrome://inspect. They cannot see the JS console. They cannot read truncated error text.**

### 10.1 The error bridge pattern

Inject a bootstrap script via `injectedJavaScriptBeforeContentLoaded` that:
1. Captures `window.onerror` (runtime errors)
2. Captures `unhandledrejection` (unhandled promise rejections)
3. Wraps `console.error`
4. Stringifies object arguments via `JSON.stringify`
5. Includes stack traces when available
6. Forwards everything to native via `window.ReactNativeWebView.postMessage(JSON.stringify({kind, text}))`

On the native side:
- `onMessage` handler parses the JSON and appends to a `jsErrors` state array (cap at 50 entries).
- A red banner at the bottom of the screen shows the latest error.
- Tapping the banner opens a full-screen `<Modal>` with a `<ScrollView>` showing ALL errors in monospace.
- "Copy all" button uses `expo-clipboard` to copy the entire log to the system clipboard.

### 10.2 Why this matters

Without this, when something breaks you get one of:
- A blank screen
- A frozen UI
- A vague "something went wrong" with no details

With this, the user can tap → read → copy → paste into your chat. You get the actual stack trace. Diagnosis time drops from "I have no idea" to "oh it's the getAssetPrefix invariant" in one message.

### 10.3 Implementation reference

See `/home/z/my-project/expo-app/App.tsx`. Search for:
- `JS_BRIDGE_BOOTSTRAP` — the injected script
- `ErrorModal` — the full-screen error console component
- `pushError` — the state update function
- `errorBanner` styles — the red banner at the bottom

## 12. Known limitations of the WebView approach

Be honest with the user about these:

1. **iOS limitations:** Service workers don't work in `file://` origins on iOS WebKit. If the PWA depends on SW for offline caching, it'll work on Android but not iOS. (The app still works offline because all assets are local files; it's just the SW-level caching that's missing.)

2. **File picker may not work:** `<input type="file">` inside a WebView requires extra native wiring (`WebChromeClient.onShowFileChooser`) that `react-native-webview` does support but doesn't enable by default. If the user reports "I can't pick files", that's a separate fix — likely needs `setSupportMultipleWindows={true}` and a custom `WebChromeClient` subclass, OR migrating the file-picker feature to a native module.

3. **Performance:** WebView startup is slower than a native app (~1-2 seconds for the first paint). After that, performance depends on the PWA itself.

4. **APK size:** You're shipping a full browser engine + the entire PWA. Expect 50-80 MB minimum. There's no way around this with the WebView approach.

5. **No push notifications, no deep linking, no native APIs** unless you add native bridges. The WebView is a sandbox.

6. **Debug builds vs release builds behave differently.** Always test the release APK, not just Expo Go. Some issues (like the getAssetPrefix invariant) only manifest in release because of how Turbopack minifies.

## 17. Lessons learned (the painful way)

1. **The Turbopack runtime is the enemy.** It hardcodes path assumptions that work for `http(s)://` origins but break for `file://`. Every Next.js version may shuffle these checks around — search for `indexOf("_next")` and `Expected document.currentScript` in the bundle if hydration fails.

2. **Disk space is the silent killer.** A 10 GB sandbox is barely enough. Clean early, clean often. The transforms cache is the biggest single chunk you can safely delete.

3. **Don't trust `nohup` in sandboxes.** Use `setsid` or run in foreground with a long tool timeout. Backgrounded processes die silently when the parent shell exits.

4. **New Architecture = New Headaches.** Disable it. You don't need Fabric or TurboModules for a WebView wrapper. The codegen step alone will eat 30 minutes of your life.

5. **The user can't debug.** Build the debugging UI into the app. The error bridge + tappable modal + copy-to-clipboard pattern is non-negotiable for shipping WebView apps to non-technical users.

6. **Static export first, questions later.** Any Next.js feature that requires a server (API routes, SSR, ISR, image optimization) must be eliminated before you can wrap it. Audit early.

7. **Bump the bundle version on every web change.** Otherwise the app reuses the stale unpacked files from `documentDirectory` and your fix doesn't ship.

8. **The Expo Go URL is a lie in sandboxes.** ngrok connects, but the dev server dies. Build the APK. Always.

9. **`expo-asset` for folders is fragile.** Metro doesn't know how to handle `.js` files in arbitrary folders. Use the data-module approach (`web-bundle.ts`).

10. **Document everything in worklog.md.** The next agent (or you in a new session) will thank you. Append, don't overwrite.

## 18. Haptics

> **Package:** `expo-haptics` (already in Expo Go, no native build needed)
> **Install:** `npx expo install expo-haptics`
> **Platform:** iOS full support · Android limited (maps to system vibration patterns)

### 18.1 Tap haptics — `S`

**What:** Fire `Haptics.impactAsync(ImpactFeedbackStyle.Light)` on every native button/tab press.

**Why:** Native apps respond with a tiny physical tick when you tap. Without it, taps feel "dead" and users tap again thinking the first one didn't register. WebView buttons get this for free via the OS, but your native chrome (error banner, modal buttons, future tab bar) doesn't.

**How:** Wrap every `onPress` in a helper:

```ts
import * as Haptics from "expo-haptics";
import { useSettingsStore } from "./settings";  // your zustand store

export function useHaptic() {
  const hapticsEnabled = useSettingsStore((s) => s.hapticsEnabled);
  return (style: Haptics.ImpactFeedbackStyle = Haptics.ImpactFeedbackStyle.Light) => {
    if (!hapticsEnabled) return;
    Haptics.impactAsync(style).catch(() => {});
  };
}

// Usage:
const haptic = useHaptic();
<TouchableOpacity onPress={() => { haptic(); doThing(); }}>
```

**Gotchas:**
- Don't fire haptics on scroll/drag events — the rapid-fire calls overwhelm the taptic engine and feel terrible.
- iOS simulator doesn't render haptics. Test on a real device.

### 18.2 Selection haptics — `S`

**What:** `Haptics.selectionAsync()` when changing radio buttons, segmented controls, theme toggles, font-size steppers.

**Why:** Selection haptics are subtler than impact haptics — a tiny "tick" that says "the value changed" without the heavier "thunk" of a button press. Standard iOS pattern; users expect it.

```ts
const handleThemeToggle = () => {
  haptic(Haptics.ImpactFeedbackStyle?.Selection ?? Haptics.ImpactFeedbackStyle.Light);
  // ... actually, selection haptics need their own call:
  Haptics.selectionAsync();
  setTheme(theme === "dark" ? "light" : "dark");
};
```

Note: `expo-haptics` exposes `selectionAsync()` directly. Use it for any value-change interaction.

### 18.3 Success/error haptics — `S`

**What:** `Haptics.notificationAsync(NotificationFeedbackType.Success)` when a lesson compiles successfully or GitHub sync completes; `NotificationFeedbackType.Error` on parse failure.

**Why:** Pairs with toast messages to reinforce outcomes. The success pattern is a distinctive "rising" double-tick that users immediately learn to read as "yes!". The error pattern is a heavy buzz.

```ts
async function importLesson(source: string) {
  try {
    const lesson = parse(source);
    if (lesson.errors.length === 0) {
      await Haptics.notificationAsync(Haptics.NotificationFeedbackType.Success);
      Toast.show("Lesson compiled.");
    } else {
      await Haptics.notificationAsync(Haptics.NotificationFeedbackType.Error);
      Toast.show(`Compiled with ${lesson.errors.length} errors.`);
    }
  } catch (e) {
    await Haptics.notificationAsync(Haptics.NotificationFeedbackType.Error);
    Toast.show(`Parse failed: ${e.message}`);
  }
}
```

**Gotcha:** These are the most disruptive haptic patterns. Use sparingly — once per major action, not per keystroke.

### 18.4 Long-press haptics — `S`

**What:** `Haptics.impactAsync(ImpactFeedbackStyle.Medium)` when a long-press is registered (e.g. on a lesson card to open the context menu).

**Why:** Confirms the long-press was detected before the menu appears. Without it, there's an awkward "did I hold long enough?" moment.

```tsx
<TouchableOpacity
  onLongPress={() => {
    Haptics.impactAsync(Haptics.ImpactFeedbackStyle.Medium);
    openContextMenu(lesson);
  }}
  delayLongPress={400}
>
```

**Gotcha:** Pair with `delayLongPress={400}` — the default 500ms feels sluggish.

### 18.5 Question-answer haptics — `M`

**What:** Inside the WebView's lesson player, fire haptics at three moments: (a) light tick when selecting an answer, (b) success pattern when answered correctly, (c) warning pattern when answered incorrectly.

**Why:** Pedagogically reinforces the BKT feedback loop. The haptic is faster than visual feedback and creates a stronger memory association.

**How:** Bridge from web to native via `postMessage`:

In the WebView's React code (`work/src/components/clil/question-block.tsx`):
```ts
function haptic(kind: "select" | "correct" | "incorrect") {
  if (window.ReactNativeWebView) {
    window.ReactNativeWebView.postMessage(JSON.stringify({ kind: "haptic", haptic: kind }));
  }
}
```

In `App.tsx`'s `onMessage` handler:
```ts
if (msg.kind === "haptic") {
  const map = {
    select: () => Haptics.selectionAsync(),
    correct: () => Haptics.notificationAsync(Haptics.NotificationFeedbackType.Success),
    incorrect: () => Haptics.notificationAsync(Haptics.NotificationFeedbackType.Warning),
  };
  map[msg.haptic]?.();
}
```

**Gotcha:** The bridge adds ~50ms latency. For "select" feedback, that's fine; for "correct"/"incorrect" it's also fine because the user expects a beat between submitting and seeing the result.

### 18.6 Rigorous mode (haptics off toggle) — `S`

**What:** A Settings toggle "Haptic feedback" that disables all haptics. Persisted in AsyncStorage. Off by default for users with motor sensitivity or who simply find haptics annoying.

**Why:** Accessibility + user preference. Required by App Store review if haptics are prominent.

**How:**

```ts
// settings store (zustand + persist)
interface SettingsState {
  hapticsEnabled: boolean;
  setHapticsEnabled: (v: boolean) => void;
}

export const useSettingsStore = create<SettingsState>()(
  persist(

```

The `useHaptic()` hook from §18.1 already checks this flag. Just wire the Settings UI:

```tsx

```

### 19.1 Swipe-back navigation — `M`

**What:** Edge-swipe from the left edge to go back from the lesson player to the library. Standard iOS pattern, increasingly common on Android.

**Why:** Once the user is in a lesson, the only way back is the small `<` button in the top-left. An edge-swipe is faster, more natural, and matches the OS-level pattern users already know.

**How:** Wrap the lesson player in a `GestureHandlerRootView` and add a pan gesture from the left edge:

```ts
// index.ts — first line
```

### 19.2 Pull-to-refresh on library — `S`

**What:** Pull-down on the library list to re-sync GitHub libraries.

**Why:** Universally understood gesture for "refresh". The WebView's built-in `pullToRefreshEnabled` flag handles the visual indicator; you just need to wire it to the sync action.

**How:** Already enabled via `pullToRefreshEnabled={true}` on the WebView. To actually trigger sync, listen for the pull event:

### 19.3 Pinch-zoom control — `M`

**What:** Allow pinch-zoom inside lesson content for accessibility, but disable it on UI chrome (headers, tabs, buttons) to prevent accidental zoom when tapping.

**Why:** Default WebView behavior is either "always allow zoom" (makes buttons feel weird) or "never allow" (fails accessibility). Selective control is the right answer.

**How:** Inject CSS into the WebView based on which page section is active:

```css
/* In your PWA's globals.css */
.ui-chrome, .ui-chrome * {
  touch-action: manipulation; /* disables double-tap zoom, allows tap */
  -webkit-user-select: none;
}
.lesson-body, .lesson-body * {
  touch-action: pan-x pan-y pinch-zoom;
}
```

And set the viewport meta in the PWA's layout.tsx:
```tsx
export const viewport: Viewport = {
  width: "device-width",
  initialScale: 1,
  maximumScale: 5,  // allow up to 5x zoom for accessibility
  userScalable: true,
};
```

**Gotcha:** `maximumScale=1` (the common "disable zoom" setting) violates WCAG 1.4.4. Don't do it globally.

### 19.4 Double-tap zoom — `S`

**What:** Double-tap inside lesson body to toggle between 1x and 1.5x zoom.

**Why:** Faster than pinch for one-handed reading. Standard mobile-browser pattern.

**How:** Pure JS in the PWA side:

## 20. Navigation & back button

> The hardware back button on Android is the most common source of "the app quit unexpectedly" complaints. Get this right.

### 20.1 Double-tap to exit — `S`

**What:** When at root screen, hardware back shows a toast "Press back again to exit". Only exits on second press within 2 seconds.

**Why:** Prevents accidental quits when the user is just putting the phone in their pocket or shifting grip. Standard pattern in WhatsApp, Telegram, etc.

```tsx
const [backPressedOnce, setBackPressedOnce] = useState(false);

useEffect(() => {
  const sub = BackHandler.addEventListener("hardwareBackPress", () => {
    if (canGoBackRef.current) {
      webViewRef.current?.goBack();
      return true;
    }
    if (backPressedOnce) {
      BackHandler.exitApp();
      return true;
    }
    setBackPressedOnce(true);
    ToastAndroid.show("Press back again to exit", ToastAndroid.SHORT);
    setTimeout(() => setBackPressedOnce(false), 2000);
    return true;
  });
  return () => sub.remove();
}, [backPressedOnce]);
```

### 20.2 Exit confirm dialog — `S`

**What:** Alternative to §20.1 — show a native `Alert.alert` "Exit CLIL Engine?" with Yes/No when back is pressed at root.

**Why:** More explicit than a toast. Some users prefer this; some apps let the user choose which mode in Settings.

```tsx
if (!canGoBackRef.current) {
  Alert.alert("Exit CLIL Engine?", "Are you sure you want to quit?", [
    { text: "Cancel", style: "cancel" },
    { text: "Exit", style: "destructive", onPress: () => BackHandler.exitApp() },

  return true;
}
```

### 20.3 In-lesson back confirm — `S`

**What:** If the user is mid-lesson with unsaved progress (e.g. answered questions, opened scenes), pressing back asks "Leave lesson? Progress will be saved."

**Why:** Prevents the gut-punch of accidentally swiping back and losing your place in a 20-scene lesson.

**How:** The PWA tracks "dirty" state (any scene visited, any answer submitted). Bridge to native:

```ts
// In the PWA's store:
useClilStore.subscribe((state) => {
  const isDirty = state.activeLessonId && state.sessions[state.activeLessonId]?.openedScenes.length > 0;
  window.ReactNativeWebView?.postMessage(
    JSON.stringify({ kind: "dirty-state", dirty: isDirty })

```

In native `onMessage`:
```ts
if (msg.kind === "dirty-state") {
  dirtyRef.current = msg.dirty;
}
```

In back handler:
```ts
if (inLesson && dirtyRef.current) {
  Alert.alert("Leave lesson?", "Your progress will be saved.", [
    { text: "Stay", style: "cancel" },
    { text: "Leave", onPress: () => webViewRef.current?.goBack() },
  ]);
  return true;
}
```

### 20.4 Predictive back gesture (Android 13+) — `M`

**What:** Support Android 13's predictive back gesture — the user swipes from the edge and sees a preview of where they're going.

**Why:** Future-proofing. Android 14+ makes predictive back the default behavior. Apps that don't support it feel "old".

**How:** Add to `android/app/src/main/AndroidManifest.xml`:
```xml
<application
  android:enableOnBackInvokedCallback="true"
  ...>
```

Then use `BackHandler` with multiple listeners that respect priority. The predictive back animation works automatically once the manifest flag is set, but your back-handler priority must be correct: WebView history → modal close → tab navigation → exit confirm.

**Gotcha:** This requires re-testing the entire back-button flow. Predictive back doesn't play well with `BackHandler.exitApp()` — the gesture animation expects to go to the home screen.

### 20.5 Deep linking — `M`

**What:** `clil://lesson/{id}` URLs that open a specific lesson directly. Useful for sharing lesson links from chat or email.

**Why:** Without deep links, the only way to share a lesson is to share the `.clil` file. With links, users can share a reference to a specific lesson that opens directly into the player.

**How:**

1. Configure the scheme in `app.json`:
```json
"scheme": "clilengine"
```

2. Add intent filter (auto-generated by Expo prebuild for the scheme).

3. Use `expo-linking` to parse incoming URLs:
```ts
import * as Linking from "expo-linking";

useEffect(() => {
  const handleDeepLink = ({ url }: { url: string }) => {
    const parsed = Linking.parse(url);
    if (parsed.path?.startsWith("lesson/")) {
      const lessonId = parsed.path.split("/")[1];
      // Bridge to WebView
      webViewRef.current?.injectJavaScript(
        `window.__CLIL_OPEN_LESSON__("${lessonId}");`
      );
    }
  };
  Linking.addEventListener("url", handleDeepLink);
  Linking.getInitialURL().then((url) => url && handleDeepLink({ url }));
}, []);
```

4. In the PWA, expose `window.__CLIL_OPEN_LESSON__`:
```ts
window.__CLIL_OPEN_LESSION__ = (id: string) => {
  useClilStore.getState().openLesson(id);
};
```

### 20.6 Tab history stack — `M`

**What:** Each bottom tab maintains its own back stack — switching tabs preserves scroll position and lesson-open state.

**Why:** Standard native-app behavior. Currently the WebView has a single history stack, so switching tabs loses context.

**How:** Two options:

**Option A (lazy):** Keep a single WebView but inject JS to save/restore state per tab:
```ts
// On tab switch:
webViewRef.current?.injectJavaScript(`
  window.__CLIL_SAVE_TAB_STATE__("library");
  window.__CLIL_RESTORE_TAB_STATE__("mastery");
`);
```

**Option B (proper):** Render multiple WebViews, one per tab, keep them all mounted but toggle visibility. Heavy on memory but cleaner.

Recommend Option A for memory-constrained devices.

## 21. Storage & filesystem

> **Packages:** `expo-file-system`, `expo-document-picker`, `expo-media-library`
> **Android 11+ caveat:** Scoped Storage changes mean apps can't write to arbitrary paths. Use the Storage Access Framework (SAF) via DocumentPicker for user-chosen folders.

### 21.1 Custom storage folder — `M`

**What:** Let the user pick a folder (via SAF / DocumentPicker) where `.clil` files AND exports live. Persist the URI with permissions across restarts.

**Why:** Lets users keep their lessons on a cloud-synced folder (Drive, Syncthing, Nextcloud) for cross-device sync without a backend.

**How:**

```ts
import * as DocumentPicker from "expo-document-picker";
import * as FileSystem from "expo-file-system";
import * as IntentLauncher from "expo-intent-launcher";

async function pickStorageFolder() {
  const result = await DocumentPicker.getDocumentAsync({
    copyToCacheDirectory: false,
  });
  if (result.canceled) return;
  const uri = result.assets[0].uri;  // content:// URI

  // Persist permission so we can use it after restart
  if (Platform.OS === "android") {
    const perms = await FileSystem.StorageAccessFramework.requestDirectoryPermissionsAsync(uri);
    if (!perms.granted) return;
    await AsyncStorage.setItem("storageFolderUri", perms.directoryUri);
  } else {
    await AsyncStorage.setItem("storageFolderUri", uri);
  }
}

async function writeFileToStorage(name: string, content: string) {
  const folderUri = await AsyncStorage.getItem("storageFolderUri");
  if (!folderUri) throw new Error("No storage folder set");

  if (Platform.OS === "android") {
    // SAF write
    const fileUri = await FileSystem.StorageAccessFramework.createFileAsync(
      folderUri, name, "text/plain"
    );
    await FileSystem.writeAsStringAsync(fileUri, content);
  } else {
    await FileSystem.writeAsStringAsync(`${folderUri}/${name}`, content);
  }
}
```

**Gotchas:**
- SAF URIs need `takePersistableUriPermission` (handled by `requestDirectoryPermissionsAsync`).
- iOS doesn't have SAF — use the Files app via `UIDocumentPickerViewController`.
- Persist the URI in AsyncStorage, not the file path — the URI is stable across reboots.

### 21.2 Auto-export progress — `S`

**What:** Export mastery progress + lesson state as JSON to the chosen storage folder on every session end (or every 5 minutes during a session).

**Why:** Crash recovery + cross-device sync via cloud drive. If the app crashes or the phone dies, the user's progress is on their Drive.

**How:**

```ts
// On AppState change to "inactive" or "background":
AppState.addEventListener("change", (state) => {
  if (state === "background") {
    webViewRef.current?.injectJavaScript(`
      window.ReactNativeWebView?.postMessage(JSON.stringify({
        kind: "export-state",
        data: JSON.stringify(useClilStore.getState())
      }));
    `);
  }
});

// In onMessage:
if (msg.kind === "export-state") {
  const filename = `clil-progress-${new Date().toISOString().slice(0,10)}.json`;
  await writeFileToStorage(filename, msg.data);
}
```

**Gotcha:** Don't export more often than every 5 minutes — JSON serialization of the full store can be 100KB+ on a heavy-use app.

### 21.3 Import from folder (file watcher) — `L`

**What:** Watch a folder for new `.clil` files and auto-import them.

**Why:** User drops a new `.clil` file in their Drive folder → it appears in the app automatically. Magic.

**How:** Two parts:

1. **On foreground:** Scan the storage folder for `.clil` files not yet imported.
2. **Background:** Use `expo-background-fetch` or `expo-task-manager` to scan periodically (every 15 min on Android, ~10 min on iOS if configured).

```ts
import * as BackgroundFetch from "expo-background-fetch";
import * as TaskManager from "expo-task-manager";

const IMPORT_TASK = "import-new-clil-files";

TaskManager.defineTask(IMPORT_TASK, async () => {
  const folderUri = await AsyncStorage.getItem("storageFolderUri");
  if (!folderUri) return BackgroundFetch.Result.NoData;

  const files = await FileSystem.readDirectoryAsync(folderUri);
  const clilFiles = files.filter(f => f.endsWith(".clil"));
  const imported = await AsyncStorage.getItem("importedFiles") || "[]";
  const importedSet = new Set(JSON.parse(imported));

  for (const file of clilFiles) {
    if (!importedSet.has(file)) {
      const content = await FileSystem.readAsStringAsync(`${folderUri}/${file}`);
      // Bridge to WebView to import
      webViewRef.current?.injectJavaScript(
        `window.__CLIL_IMPORT_FILE__("${file}", ${JSON.stringify(content)});`
      );
      importedSet.add(file);
    }
  }
  await AsyncStorage.setItem("importedFiles", JSON.stringify([...importedSet]));
  return BackgroundFetch.Result.NewData;
});

// Register on app start:
BackgroundFetch.registerTaskAsync(IMPORT_TASK, {
  minimumInterval: 60 * 15, // 15 min
  stopOnTerminate: false,
  startOnBoot: true,
});
```

**Gotchas:**
- iOS background fetch is unreliable (the OS decides when to run it).
- Android Doze mode can delay fetches by hours.
- Don't import files that are already in the store — track by filename + content hash.

### 21.4 Storage usage panel — `S`

**What:** Settings panel showing bytes used by: lessons, mastery, web bundle, cached images. With "Clear cache" and "Export all data" buttons.

**Why:** Transparency. Users want to know why the app is using 200MB. Also helps diagnose storage-full issues.

**How:**

```tsx
async function getStorageBreakdown() {
  const webDir = `${FileSystem.documentDirectory}web/`;
  const webInfo = await FileSystem.getInfoAsync(webDir, { size: true });

  // Read AsyncStorage keys and sum their lengths
  const keys = await AsyncStorage.getAllKeys();
  let asyncStorageBytes = 0;
  for (const key of keys) {
    const val = await AsyncStorage.getItem(key);
    asyncStorageBytes += (val?.length || 0) + key.length;
  }

  return {
    webBundle: webInfo.size || 0,
    asyncStorage: asyncStorageBytes,
    cache: await getCacheSize(),
    total: (webInfo.size || 0) + asyncStorageBytes,
  };
}

// Render as a list with progress bars and a "Clear cache" button
```

### 21.5 Scoped storage compatibility — `M` (documentation)

**What:** Document the Android 11+ scoped storage workarounds.

**Why:** Android 11 introduced Scoped Storage, which fundamentally changed how apps access files. If you don't handle this correctly, file pickers and exports break on modern Android.

**Key rules:**
- Apps can read/write freely to their own `getExternalFilesDir()` (no permission needed).
- To access other folders, use SAF (Storage Access Framework) — `expo-document-picker` with `copyToCacheDirectory: false`.
- Persist URI permissions with `requestDirectoryPermissionsAsync`.
- For MediaStore (saving images/videos to Gallery), use `expo-media-library`.
- Don't request `READ_EXTERNAL_STORAGE` / `WRITE_EXTERNAL_STORAGE` — deprecated on Android 13+.

**Manifest changes** (auto-handled by Expo, but verify):
```xml
<application
  android:requestLegacyExternalStorage="false"
  ...>
```

**Gotcha:** File paths like `/sdcard/Download/foo.clil` no longer work. Use `content://` URIs everywhere.

## 22. Permissions

> **Package:** `expo-permissions` (deprecated — use module-specific permission functions now, e.g. `expo-notifications.getPermissionsAsync()`)

### 22.1 Runtime permission flow (just-in-time) — `S`

**What:** Request permissions only when the user actually triggers the feature, with a rationale dialog if previously denied.

**Why:** Asking for permissions on first launch (the old pattern) is hostile and gets you 1-star reviews. Just-in-time requests have ~3x higher grant rates.

**How:** Pattern:

```ts
async function requestPermissionWithRationale<T>(
  check: () => Promise<PermissionStatus>,
  request: () => Promise<PermissionStatus>,
  rationale: string,
  onGranted: () => T | Promise<T>,
): Promise<T | null> {
  const current = await check();
  if (current.status === "granted") return onGranted();

  if (current.status === "denied" && current.canAskAgain) {
    const result = await request();
    if (result.status === "granted") return onGranted();
  } else if (current.status === "denied" && !current.canAskAgain) {
    // User selected "Don't ask again" — show rationale + settings link
    Alert.alert("Permission needed", rationale, [
      { text: "Cancel", style: "cancel" },
      { text: "Open settings", onPress: () => Linking.openSettings() },
    ]);
  }
  return null;
}

// Usage:
await requestPermissionWithRationale(
  () => Notifications.getPermissionsAsync(),
  () => Notifications.requestPermissionsAsync(),
  "CLIL Engine needs notifications to remind you of your daily practice.",
  () => scheduleDailyReminder(),
);
```

### 22.2 Settings deep-link — `S`

**What:** Button "Open app settings" that deep-links to the system settings page for the app.

**Why:** When a user has denied a permission with "Don't ask again", the only way to grant it is to send them to settings.

```ts
import { Linking } from "react-native";

if (Platform.OS === "ios") {
  Linking.openURL("app-settings:");
} else {
  // expo-application or IntentLauncher

}
```

**Gotcha:** `expo-linking.openSettings()` does this on iOS, but Android needs the IntentLauncher approach. Or just use `Linking.openSettings()` from React Native core (works on both).

### 22.3 Notification permission (Android 13+) — `S`

**What:** On Android 13+, request `POST_NOTIFICATIONS` at first relevant moment with explanation.

**Why:** Android 13 requires explicit permission for notifications. Without it, your notifications silently no-op.

**How:** Same pattern as §22.1, but specifically for notifications:

```ts
import * as Notifications from "expo-notifications";

async function ensureNotificationPermission() {
  const { status } = await Notifications.getPermissionsAsync();

  return newStatus === "granted";
}
```

**Gotcha:** Don't request on app launch. Wait until the user enables a feature that needs notifications (e.g. toggling "Daily reminder" in Settings).

### 22.5 Foreground service (long syncs) — `L`

**What:** Foreground service + permission for long-running GitHub syncs so they survive backgrounding. Shows a persistent notification during sync.

**Why:** Android 12+ kills background long-running tasks. A foreground service keeps the sync alive and shows the user what's happening.

**How:** This requires native code or a library like `expo-foreground-actions` (community package, may need prebuild + manual linking).

```ts
import { startForegroundAction, stopForegroundAction } from "expo-foreground-actions";

await startForegroundAction({
  taskName: "CLIL Sync",
  taskTitle: "Syncing lessons…",
  taskDesc: "Pulling from GitHub",
  taskIcon: { name: "ic_notification", iconType: "drawable" },
  color: "#7dd3fc",
  linkingURI: "clilengine://",

}
```

**Gotcha:** Add `FOREGROUND_SERVICE` permission to AndroidManifest. Don't use this for short operations (<10s) — just use a regular `setTimeout`.

### 22.6 Battery opt-out prompt — `M`

**What:** Detect if battery optimization is killing background syncs, prompt user to disable it for the app.

**Why:** Doze mode on Android can delay background fetches by hours. If the user complains "auto-sync doesn't work", this is usually why.

**How:**

```ts
import { IntentLauncher } from "expo-intent-launcher";
```

## 23. Screen & power

> **Packages:** `expo-keep-awake`, `expo-brightness`, `expo-battery`

### 23.1 Keep screen awake during lessons — `S`

**What:** `expo-keep-awake` — keep screen on during a lesson (deactivate on pause). Toggle in Settings.

**Why:** Education apps need this — nothing worse than the screen dimming mid-question. Standard pattern in Duolingo, Khan Academy, etc.

```ts
import { useKeepAwake, activateKeepAwakeAsync, deactivateKeepAwake } from "expo-keep-awake";

// In the lesson player component:
useKeepAwake();  // auto-activates on mount, deactivates on unmount

// Or manual control:
if (settings.keepAwakeDuringLessons && inLesson) {
  activateKeepAwakeAsync("lesson-session");
} else {
  deactivateKeepAwake("lesson-session");
}
```

**Gotcha:** Tag your keep-awake sessions (`"lesson-session"`) so multiple keep-awake activations don't conflict.

### 23.2 Brightness boost — `M`

**What:** Temporarily boost screen brightness during lesson reading, restore on exit.

**Why:** Useful for outdoor use. Some reading apps (Kindle) do this.

```ts
import * as Brightness from "expo-brightness";

async function boostBrightness() {
  const original = await Brightness.getBrightnessAsync();
  await Brightness.setBrightnessAsync(1.0);  // 100%
  return original;
}

// On lesson exit:
const original = await boostBrightness();
// ... later ...
await Brightness.setBrightnessAsync(original);
```

**Gotcha:** On Android, `setBrightnessAsync` only affects the app's window brightness, not system brightness. That's actually what you want — don't change the user's system setting. Requires `WRITE_SETTINGS` permission on Android.

### 23.3 Battery-saver mode — `M`

**What:** Detect low battery (<15%) and automatically reduce animations, disable haptics, dim background.

**Why:** Extends battery life when the user needs it most. Subtle, appreciated.

```ts
import * as Battery from "expo-battery";

const batteryLevel = await Battery.getBatteryLevelAsync();  // 0..1
const lowPower = Battery.isLowPowerModeEnabled?.() || false;  // iOS only
const isLow = batteryLevel < 0.15 || lowPower;

if (isLow) {
  settings.setHapticsEnabled(false);
  settings.setAnimationsEnabled(false);
  // Bridge to WebView to disable framer-motion springs
  webViewRef.current?.injectJavaScript(`window.__CLIL_LOW_POWER__ = true;`);
}
```

**Gotcha:** On Android, there's no clean API to detect system battery-saver mode. Use the level threshold (<15%) as a proxy. Listen to `Battery.addEventListener("batteryLevelChange", ...)` for live updates.

### 23.4 Dark mode sync — `S`

**What:** App theme follows system dark mode by default; setting to override. Bridge the native appearance to the WebView.

**Why:** Users expect the app to honor their system theme. Currently the PWA has its own theme toggle — that's fine, but it should default to the system value.

```ts
import { useColorScheme } from "react-native";

const systemTheme = useColorScheme();  // "light" | "dark" | null
const userOverride = useSettingsStore(s => s.themeOverride);  // "system" | "light" | "dark"
const effective = userOverride === "system" ? systemTheme : userOverride;

// Bridge to WebView on every change:
useEffect(() => {
  webViewRef.current?.injectJavaScript(
    `window.__CLIL_SET_THEME__("${effective}");`
  );
}, [effective]);

// In the PWA:
window.__CLIL_SET_THEME__ = (theme) => {
  document.documentElement.classList.toggle("dark", theme === "dark");
};
```

### 23.5 Auto-pause on background — `S`

**What:** When app goes to background, pause any running simulation in the lesson. Resume on foreground.

**Why:** Saves battery + CPU. A lesson with a JS simulation running can drain 5%/hour in the background.

```ts
import { AppState } from "react-native";

useEffect(() => {
  const sub = AppState.addEventListener("change", (state) => {
    if (state === "background") {
      webViewRef.current?.injectJavaScript(`
        document.querySelectorAll('iframe[srcdoc]').forEach(f => {
          f.dataset.savedSrcDoc = f.srcdoc;
          f.srcdoc = '';
        });
      `);
    } else if (state === "active") {
      webViewRef.current?.injectJavaScript(`
        document.querySelectorAll('iframe[data-saved-srcdoc]').forEach(f => {
          f.srcdoc = f.dataset.savedSrcDoc;
          delete f.dataset.savedSrcDoc;
        });
      `);
    }
  });
  return () => sub.remove();
}, []);
```

**Gotcha:** Don't unload the entire WebView — just the iframes (which is where simulations live). Otherwise the user's scroll position and lesson state are lost.

### 25.3 Streak notifications — `M`

**What:** Daily-streak tracking: notify "You're on a 7-day streak! Keep it up."

**Why:** Gamifies learning. Stored locally (no analytics, no server).

**How:** Track `lastActiveDate` in AsyncStorage. On app open, if it's a new day since `lastActiveDate`, increment `streak`. If it's been >2 days, reset to 1.

```ts
async function updateStreak() {
  const today = new Date().toDateString();
  const last = await AsyncStorage.getItem("lastActiveDate");
  const streak = parseInt(await AsyncStorage.getItem("streak") || "0");

  if (last === today) return;  // already counted

  const yesterday = new Date(Date.now() - 86400000).toDateString();
  const newStreak = last === yesterday ? streak + 1 : 1;

  await AsyncStorage.setItem("lastActiveDate", today);
  await AsyncStorage.setItem("streak", String(newStreak));

  // Fire milestone notifications
  if ([3, 7, 14, 30, 60, 100, 365].includes(newStreak)) {
    await Notifications.scheduleNotificationAsync({
      content: {
        title: `${newStreak}-day streak! 🔥`,
        body: "Keep practicing — you're building a habit.",
      },
      trigger: null,
    });
  }
}
```

## 26. WebView performance & UX

### 26.1 Pre-warm WebView — `M`

**What:** Initialize the WebView in the background on the splash screen so the first lesson opens instantly.

**Why:** WebView cold start is ~500ms. Pre-warming hides this behind the splash screen.

**How:** Render the WebView hidden during the splash state, then reveal it when ready:

```tsx
const [webviewReady, setWebviewReady] = useState(false);

      <WebView
        ref={webViewRef}
        source={{ uri: INDEX_FILE }}
        onLoad={() => setWebviewReady(true)}
        ...
      />
    </View>
    {!webviewReady && <SplashScreen />}
  </>
);
```

### 26.2 Cache bundle unpack — `S`

**What:** Skip re-unpacking on every startup by stamping a hash of `WEB_BUNDLE` content (not just version string).

**Why:** If you forget to bump `BUNDLE_VERSION` after a web change, the current code still re-unpacks because the marker string doesn't match. With a content hash, the check is always correct.

```ts
// Generate a hash of the bundle at build time:
// In scripts/build-web-bundle.py, add:
import hashlib
h = hashlib.sha256(json.dumps(files).encode()).hexdigest()[:16]
# Write it as a separate export:
f.write(f"\nexport const WEB_BUNDLE_HASH = \"{h}\";\n")

// In App.tsx:
import { WEB_BUNDLE, WEB_BUNDLE_HASH } from "./src/web-bundle";

async function ensureWebBundle() {
  const marker = await FileSystem.readAsStringAsync(VERSION_MARKER);
  if (marker === WEB_BUNDLE_HASH) return;  // skip unpack
  // ... unpack ...
  await FileSystem.writeAsStringAsync(VERSION_MARKER, WEB_BUNDLE_HASH);
}
```

### 26.3 Disable WebView bounce (iOS) — `S`

**What:** Suppress the iOS rubber-band scroll bounce on the WebView root.

```tsx
<WebView
  ...
  bounces={false}
  scrollEnabled={false}  // the PWA handles its own scrolling
/>
```

**Gotcha:** `scrollEnabled={false}` means the WebView itself doesn't scroll, but internal `<div>`s with `overflow: auto` do. This is what you want — the native scroll view bounces, the inner divs don't.

### 26.4 Native splash handoff — `M`

**What:** Keep the native splash screen visible until the WebView fires `onLoad`.

**Why:** Eliminates the white flash between native splash and PWA first paint.

**How:** Use `expo-splash-screen`:

```ts
import * as SplashScreen from "expo-splash-screen";

```

### 29.1 Share sheet (outbound) — `S`

**What:** Share a lesson source via native share sheet.

```ts
import * as Sharing from "expo-sharing";
import * as FileSystem from "expo-file-system";

async function shareLesson(lesson: { name: string; source: string }) {
  const fileUri = `${FileSystem.cacheDirectory}${lesson.name}.clil`;
  await FileSystem.writeAsStringAsync(fileUri, lesson.source);
```

Then in App.tsx, read the file and bridge to the WebView:

```ts
const initialFileUri = props.initialProps?.sharedFileUri;
```

## 30. Security & privacy

### 30.1 Screenshot block — `S`

**What:** `FLAG_SECURE` on the WebView window — prevents screenshots/screen recording.

**Why:** Useful for content-licensed lessons where the author doesn't want their material captured.

**How:** Config Plugin or modify `MainActivity.kt`:
```kotlin
override fun onCreate(savedInstanceState: Bundle?) {
  super.onCreate(savedInstanceState)
  if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.TIRAMISU) {
    window.setFlags(WindowManager.LayoutParams.FLAG_SECURE, WindowManager.LayoutParams.FLAG_SECURE)
  }
}
```

Make it a Settings toggle — users hate having screenshots blocked by default.

### 30.2 Biometric lock — `M`

**What:** `expo-local-authentication` — require FaceID/fingerprint to open the app (or to open a specific "locked" lesson).

```ts
import * as LocalAuthentication from "expo-local-authentication";

async function unlock() {
  const compatible = await LocalAuthentication.hasHardwareAsync();
  if (!compatible) return true;  // skip on devices without biometrics

  const enrolled = await LocalAuthentication.isEnrolledAsync();
  if (!enrolled) return true;

  const result = await LocalAuthentication.authenticateAsync({
    promptMessage: "Unlock CLIL Engine",
    fallbackLabel: "Use passcode",
  });
  return result.success;
}

// On app launch:
if (settings.biometricLock && !(await unlock())) {
  BackHandler.exitApp();
}
```

### 30.3 Incognito mode — `M`

**What:** Per-session flag that disables mastery tracking and clears all session state on exit.

**Why:** For users practicing without polluting their stats. Privacy feature.

**How:** Add an `incognito` flag to the store. When true:
- Don't persist `sessions` to localStorage.
- Don't fire mastery milestone notifications.
- On exit, clear `sessions[activeLessonId]`.

### 30.4 No-telemetry guarantee — `S` (documentation)

**What:** Document that the app makes zero outbound network calls except user-initiated GitHub fetches. List every domain it CAN contact.

**Why:** Privacy-conscious users (and App Store reviewers) want to know. Add a "Privacy" section in Settings.

```
Network access:
- api.github.com (only when you tap "Add library" or "Sync")
- raw.githubusercontent.com (only when syncing a library)
- No analytics. No crash reporting. No telemetry.
- All your data stays on your device.
```

### 30.5 URL allowlist — `S`

**What:** WebView should refuse to navigate to any URL except `file://`, `api.github.com`, `raw.githubusercontent.com`, `github.com`.

**Why:** Block ads, trackers, accidental link taps. Some lesson content might try to load external resources — keep it locked down.

**How:** Use `onShouldStartLoadWithRequest`:

```tsx
<WebView
  onShouldStartLoadWithRequest={(req) => {
    const url = new URL(req.url);
    const allowed = ["file", "https"];
    const allowedHosts = ["api.github.com", "raw.githubusercontent.com", "github.com", ""];
    return allowed.includes(url.protocol) && allowedHosts.includes(url.hostname);
  }}
/>
```

### 30.6 Auto-lock on background — `S`

**What:** If biometric lock is enabled, re-lock the app when it goes to background for >30 seconds.

```ts
let backgroundedAt: number | null = null;

AppState.addEventListener("change", (state) => {
  if (state === "background") {
    backgroundedAt = Date.now();
  } else if (state === "active" && backgroundedAt) {
    if (Date.now() - backgroundedAt > 30_000 && settings.biometricLock) {
      unlock();
    }
    backgroundedAt = null;
  }
});
```

### 32.3 Anti-patterns to avoid

1. **Don't ship all 69 features at once.** Pick a wave, ship it, get feedback, then iterate.
2. **Don't add features without Settings toggles.** Every behavioral change should be opt-out.
3. **Don't duplicate PWA features in native.** If the PWA already does it (e.g. dark mode toggle), bridge — don't rebuild.
4. **Don't add native features that the PWA can do.** Most "polish" features are better implemented in the PWA's React code where they're easier to iterate on.
5. **Don't add `expo-background-fetch` without testing on real Android devices.** Doze mode will make a mockery of your scheduled tasks.
6. **Don't add biometric lock without a fallback.** Devices without biometrics need a passcode option or the user gets locked out forever.

### 32.4 Testing checklist per feature

Before marking a feature done:
- [ ] Works on Android 10, 13, 14+ (emulator or device)
- [ ] Works on at least one real device (not just emulator)
- [ ] Has a Settings toggle (if behavioral)
- [ ] Has accessibility labels (if it adds UI)
- [ ] Doesn't break existing features (regression test)
- [ ] Documented in the Settings page (so users can find it)
- [ ] Worklog entry updated
